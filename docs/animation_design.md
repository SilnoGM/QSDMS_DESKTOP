# QSDMS_DESKTOP 全局动画设计方案

> 适用项目：`QSDMS_DESKTOP`  
> 适用端：`Flutter Desktop`，目标平台为 `macOS`、`Windows`  
> 技术栈：`Flutter Desktop + GetX + flutter_animate`  
> 文档目标：定义应用级动效方向、参数体系、组件使用规则与验收标准，作为后续所有页面和组件的全局动画设计依据。

## 1. 设计结论

`QSDMS_DESKTOP` 的动画风格不再采用传统企业后台常见的克制、低存在感、仅做状态切换的动效方案。后续全局动画应转向更年轻、更活泼、更具画面灵动性的消费级体验。

这不是把系统做成娱乐化界面，而是在保证数据管理效率的前提下，让界面状态变化更清晰、反馈更及时、视觉更有呼吸感。动画应服务于用户理解当前发生了什么，同时提升应用的品牌识别和使用愉悦度。

全局动效方向如下：

| 项目 | 结论 |
|---|---|
| 风格定位 | 年轻、轻快、灵动、消费级、带一点弹性 |
| 不采用 | 企业后台式克制动画、机械淡入淡出、无情绪的硬切换 |
| 核心体验 | 状态变化有路径，重点元素有跟随，反馈有弹性 |
| 主要技术 | `flutter_animate`、`AnimationController`、`SpringSimulation` |
| 默认时长 | 普通状态 `180-260ms`，关键转场 `360-500ms` |
| 默认曲线 | `Curves.easeOutCubic`，关键位移可使用弹簧模拟 |
| 动效密度 | 高频操作轻量，页面级变化更有画面感 |

## 2. 全局原则

### 2.1 动画要表达状态路径

只做“新状态出现、旧状态消失”是不够的。用户需要看到状态从哪里来、到哪里去。

推荐：

- 选中背景从旧菜单项滑向新菜单项。
- 详情面板从触发区域方向进入。
- 折叠内容从自身位置展开，而不是直接出现。
- 批量操作栏从选择行为附近浮现。

不推荐：

- 菜单选中背景直接跳到新位置。
- 弹窗没有来源关系地突然出现。
- 列表刷新时整块内容硬切。
- hover、selected、loading 只有颜色瞬间变化。

### 2.2 使用共享元素思路

当两个状态之间存在同一个视觉对象时，应优先让这个对象移动、变形或过渡，而不是让两个对象分别淡入淡出。

典型场景：

- 菜单选中胶囊。
- Tab 选中下划线或背景。
- Segmented control 选中块。
- 表格行展开的详情容器。
- 卡片进入详情页的头部信息。

共享元素能让界面更“连续”，也是本项目后续实现灵动感的基础。

### 2.3 关键变化允许更有弹性

传统企业后台常把所有动画压到 `120-180ms`，结果是界面很快但缺少生命力。本项目允许关键状态使用 `360-500ms` 的过渡，并通过曲线或弹簧让动画更自然。

适合更有弹性的场景：

- 主导航切换。
- 侧边栏展开 / 折叠。
- 页面主内容切换。
- Drawer、Popover、Modal 进入。
- 大面积筛选面板展开。

不适合过度弹性的场景：

- 文本输入。
- 表格单元格编辑。
- 高频点击按钮。
- 数据校验错误提示。
- 批量操作中的连续选择。

### 2.4 动效要有层次

同一段交互中，不同元素不应同时用同样的幅度、同样的时长移动。应形成主次：

- 主元素：承担最大位移或背景过渡。
- 辅助元素：做轻微位移、缩放或透明度变化。
- 文案和图标：以颜色、权重、轻微跟随为主。
- 装饰层：可延迟或更慢，形成画面呼吸感。

例如菜单切换中，选中背景是主动画，图标和文字只做轻微跟随，不抢动效主体。

### 2.5 动效必须可感知但不阻塞

动画可以有个性，但不能影响工作效率。

约束：

- 高频操作反馈不应超过 `260ms`。
- 页面主流程不应等待装饰动画结束才能操作。
- loading 动效应传达等待状态，不应制造焦虑。
- 动画结束前如果用户继续操作，应允许新动画接管当前状态。

## 3. Motion Token

### 3.1 时长

| Token | 建议值 | 适用场景 |
|---|---:|---|
| `instant` | `0-80ms` | 禁用动画、测试、极轻微状态同步 |
| `fast` | `120-180ms` | hover、press、focus、轻量颜色变化 |
| `normal` | `220-260ms` | 按钮反馈、列表项进入、局部展开 |
| `expressive` | `360-420ms` | 弹窗、Drawer、页面局部切换 |
| `hero` | `500ms` | 主导航选中背景、关键共享元素位移 |

当前菜单项前景跟随动画使用 `500ms`，目的是让用户能清楚感知背景从旧菜单项移动到新菜单项。后续如果某个组件的切换频率很高，应优先降到 `260-360ms`。

### 3.2 曲线

| Token | 推荐曲线 | 说明 |
|---|---|---|
| `standardOut` | `Curves.easeOutCubic` | 默认出场、状态切换 |
| `standardInOut` | `Curves.easeInOutCubic` | 双向展开 / 收起 |
| `emphasizedOut` | `SpringSimulation` | 主导航、共享背景、关键位移 |
| `softIn` | `Curves.easeInCubic` | 离场、弱化、消失 |

禁止把 `Curves.linear` 作为普通 UI 动画默认值。线性动画只适合进度条、旋转 loading、时间轴播放等连续运动。

### 3.3 位移和缩放

| 类型 | 建议值 | 适用场景 |
|---|---:|---|
| 轻微跟随位移 | `2-4px` | 菜单项、按钮、列表 hover |
| 小组件进入位移 | `6-12px` | Tooltip、Popover、表单错误提示 |
| 面板进入位移 | `16-28px` | Drawer、Filter Panel、Modal |
| 轻微缩放 | `1.01-1.03` | active、hover、press 回弹 |
| 卡片强调缩放 | `1.03-1.06` | 可点击卡片、空状态行动点 |

缩放必须克制，不能造成文字抖动、布局挤压或表格行高变化。需要缩放时优先在 `Transform` 层完成，不应改变布局尺寸。

## 4. 组件级动画规范

### 4.1 主导航和菜单

主导航是全局动效风格最重要的表达区域，应使用更灵动的消费级方案。

要求：

- 选中背景使用共享背景层，不在每个菜单项内部各自绘制选中背景。
- 切换菜单时，背景从旧菜单项位置滑向新菜单项位置。
- 背景可以使用品牌蓝渐变、透明度、阴影和轻量高光。
- 图标和文字跟随选中状态做轻微位移、缩放和颜色过渡。
- 快速连续点击时，新动画应接管当前速度，不应回到起点重播。

当前侧边栏已落地的方向：

- 使用共享选中背景。
- 跨路由切换时记录旧菜单项和新菜单项，让新页面中的背景从旧位置继续滑动。
- 背景移动使用 `SpringSimulation`。
- 菜单项前景跟随使用 `flutter_animate`，当前观察时长为 `500ms`。

### 4.2 Tab 和 Segmented Control

Tab、分段控件与菜单同属“选中态切换”，应复用共享选中层思路。

要求：

- 选中底线、胶囊或背景块从旧项滑到新项。
- 内容区域可做轻微横向位移和透明度过渡。
- 同级 Tab 切换不使用页面级大转场。
- 禁止每个 Tab 各自淡入淡出选中背景。

建议参数：

| 场景 | 时长 | 曲线 |
|---|---:|---|
| 选中块移动 | `260-420ms` | `SpringSimulation` 或 `easeOutCubic` |
| 内容切换 | `180-260ms` | `easeOutCubic` |

### 4.3 按钮和操作入口

按钮动画应轻快，不抢主流程。

要求：

- hover：颜色、阴影或轻微上浮，`120-180ms`。
- press：轻微缩小到 `0.98-0.99`，松开回弹。
- primary action 可以有更明显的 hover 光泽或阴影。
- destructive action 不使用过于活泼的弹跳，避免削弱风险感。
- disabled 状态不做吸引注意力的动画。

### 4.4 表格和列表

表格是数据管理系统的核心，动画必须帮助用户定位变化，而不是干扰扫描。

要求：

- 行 hover 使用轻量背景和微弱位移，不改变行高。
- 新增行可以从 `6-10px` 下方淡入。
- 删除行应先弱化再收起，避免数据突然消失。
- 排序、筛选、分页后不应整表大幅滑动。
- 批量选择时，批量操作栏可以从表格上方或底部自然浮现。

不推荐：

- 每一行使用大幅错峰动画。
- 表格刷新时所有行从屏幕外飞入。
- 鼠标经过表格时出现过强阴影或缩放。

### 4.5 表单和校验

表单动画应围绕“输入反馈”和“错误定位”。

要求：

- focus 状态使用边框、阴影或背景柔和过渡，`120-180ms`。
- 错误提示从字段下方轻微展开，`180-220ms`。
- 错误字段可使用一次轻微横向提醒，不做持续抖动。
- 提交中状态应让按钮、表单区域和 loading 状态同步变化。
- 表单区块展开 / 收起可用 `260-360ms`。

### 4.6 Modal、Drawer、Popover

浮层是全局动画风格的重要载体，应有明确来源和进入方向。

要求：

- Modal：背景遮罩淡入，主体从触发方向轻微上浮或缩放进入。
- Drawer：从对应边缘滑入，内容可轻微延迟出现。
- Popover：从触发点附近展开，带轻微缩放和透明度。
- 关闭时比打开更快，避免拖慢任务。
- 浮层进入后应立即可操作，不等待装饰动画结束。

建议参数：

| 组件 | 打开 | 关闭 |
|---|---:|---:|
| Modal | `300-420ms` | `180-240ms` |
| Drawer | `360-420ms` | `220-260ms` |
| Popover | `180-240ms` | `120-180ms` |

### 4.7 页面和内容区切换

桌面端页面切换不应使用移动端整页滑入式动画。全局路由可以保持无页面滑入，但页面内部内容应有过渡。

要求：

- 主框架常驻，内容区变化。
- 页面标题、工具栏、主内容可做轻微错峰进入。
- 同级模块切换时，侧边栏选中背景承担主要动效。
- 页面内容区域使用 `fade + 8-16px move` 即可。
- 不使用整页从右侧推入的移动端转场。

### 4.8 Loading、Skeleton 和空状态

等待状态应轻盈、有呼吸感，但不能过度装饰。

要求：

- Skeleton 使用柔和 shimmer，速度稳定。
- 局部 loading 优先使用小范围动效，不遮挡整页。
- 空状态可以更活泼，允许插画、轻微浮动或渐入。
- 成功反馈可以使用短促弹性，失败反馈应清晰稳定。

### 4.9 数据可视化

图表动画应帮助理解数据变化。

要求：

- 图表首次进入可以逐步生长。
- 数据更新应从旧值过渡到新值。
- tooltip 出现要跟随鼠标位置轻柔过渡。
- 不做与数据无关的循环装饰动画。

## 5. 技术落地规则

### 5.1 优先使用成熟动画能力

常规组件动画优先使用：

- `flutter_animate`
- `AnimatedContainer`
- `AnimatedOpacity`
- `AnimatedSwitcher`
- `TweenAnimationBuilder`

需要保留速度、处理连续切换或实现共享背景移动时，使用：

- `AnimationController`
- `SpringSimulation`
- `AnimatedBuilder`
- `Transform`

不应为了简单颜色或透明度变化手写复杂 controller。

### 5.2 动画不改变布局稳定性

动画原则上应发生在绘制层，而不是改变布局层。

推荐：

- `Transform.translate`
- `Transform.scale`
- `Opacity`
- `DecoratedBox` 颜色和阴影过渡

谨慎使用：

- 直接改变 `width`、`height`
- 动画过程中改变字体大小
- 动画过程中改变表格行高
- 动画过程中让文字重新换行

### 5.3 快速连续操作必须自然接管

用户连续点击菜单、Tab、筛选项时，动画不能卡顿或重置。

要求：

- 新目标应从当前动画值继续。
- 弹簧动画应尽量保留当前速度。
- 不应每次点击都从初始状态重新播放。
- 不应因为动画未完成而阻止用户继续选择。

### 5.4 可访问性和性能

动画必须尊重用户和设备状态。

要求：

- 后续应支持 reduced motion 策略。
- 列表和表格的大量 item 不应同时播放复杂动画。
- 阴影、模糊和透明叠层应控制数量。
- macOS 和 Windows 都必须验证无明显掉帧。
- 动画不能影响键盘导航和屏幕阅读器语义。

## 6. 已落地样例：侧边栏菜单

侧边栏菜单是当前全局动画方向的第一个落地点，但不是本文件的全部范围。

当前实现要点：

- `QsdmsSidebar` 使用共享选中背景层。
- `_SidebarSpringIndicator` 使用 `SpringSimulation` 驱动纵向位移。
- `SidebarSelectionMotion` 记录跨路由菜单切换的旧菜单项和新菜单项。
- `SidebarMenuItem` 使用 `flutter_animate` 处理图标和文本的轻微跟随。
- 选中背景使用品牌蓝渐变和阴影，形成更消费级的视觉重点。

后续类似组件应复用这些原则，而不是复制具体代码。

## 7. 验收标准

新增或改造组件动画时，必须满足：

- 状态变化有清晰路径，不硬切。
- 关键选中态优先使用共享元素或共享背景。
- 动效风格年轻、轻快、灵动，不回退到传统后台的机械淡入淡出。
- 高频操作不拖慢效率。
- 动画过程中布局不抖动、不溢出、不造成文字换行闪烁。
- 快速连续操作时动画能自然接管。
- 有测试覆盖关键状态变化，至少验证初始态、中间态和最终态。
- 完成前运行 `dart format --set-exit-if-changed .`、`flutter analyze`、`flutter test`。

## 8. 后续扩展

建议后续在代码层补充统一 motion token，例如：

```dart
abstract final class AppMotion {
  static const fast = Duration(milliseconds: 180);
  static const normal = Duration(milliseconds: 260);
  static const expressive = Duration(milliseconds: 420);
  static const hero = Duration(milliseconds: 500);

  static const standardOut = Curves.easeOutCubic;
  static const standardInOut = Curves.easeInOutCubic;
}
```

当页面和组件数量增加后，应把通用动画能力沉淀为公共组件或 helper，避免每个模块重复定义时长、曲线和弹簧参数。
