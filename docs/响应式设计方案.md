# QSDMS_DESKTOP 响应式设计方案

> 适用项目：`QSDMS_DESKTOP`  
> 适用端：`Flutter Desktop`，目标平台为 `macOS`、`Windows`  
> 技术栈：`Flutter Desktop + GetX + window_manager`  
> 文档目标：定义桌面端响应式与自适应规则，确保应用在不同窗口尺寸、不同屏幕分辨率和不同使用场景下稳定可用。

## 1. 背景与目标

`QSDMS_DESKTOP` 是桌面优先的企业数据管理系统，核心用户是内部运营人员。应用主要承载订单、供应商、产品、仓储、发运、财务等高频业务操作。

本项目不是移动端应用的桌面放大版，也不是普通管理后台网页的简单迁移。桌面端响应式设计的目标不是让所有内容在任意小尺寸下完整展示，而是在可接受的窗口尺寸范围内保证：

- 主流程始终可操作。
- 关键信息始终可识别。
- 高频操作始终容易触达。
- 表格、筛选、表单、详情面板不会出现不可控溢出。
- `macOS` 与 `Windows` 的窗口体验尽量一致。

当前项目已通过 `window_manager` 设置默认窗口尺寸与最小窗口尺寸：

```dart
const windowOptions = WindowOptions(
  size: Size(1280, 800),
  minimumSize: Size(1280, 800),
  center: true,
  backgroundColor: Color(0xFFFAFBFC),
  title: 'QSDMS-千树数据管理系统',
);
```

这解决了窗口被压缩到不可用的问题，但不等于完成响应式设计。真正的响应式能力需要由页面布局、业务组件、滚动策略、信息优先级和验证标准共同保证。

## 2. 设计原则

### 2.1 桌面优先

所有页面默认按照桌面端工作台体验设计。布局应优先支持：

- 鼠标点击、悬停、右键菜单。
- 键盘快捷操作。
- 高密度数据表格。
- 批量选择与批量操作。
- 多条件筛选。
- 主列表与详情联动。
- 异常优先处理。

不应先设计移动端布局，再把组件拉宽到桌面尺寸。

### 2.2 最小尺寸保底

当前推荐保持 `1280 x 800` 作为应用最小可用尺寸。低于该尺寸时，与其让界面继续压缩，不如通过 `minimumSize` 阻止窗口进一步缩小。

这样做符合企业桌面系统的实际使用场景：系统需要展示表格、筛选栏、操作栏和业务详情，不适合在过窄窗口中强行工作。

### 2.3 内容优先级明确

不同尺寸下不应简单等比例缩放所有内容，而应根据业务优先级决定展示方式。

优先级建议如下：

| 优先级 | 内容类型 | 响应式策略 |
|---|---|---|
| P0 | 当前任务、主表格核心列、主要操作按钮、状态与异常信息 | 始终展示 |
| P1 | 常用筛选项、关键统计、详情摘要 | 标准尺寸展示，紧凑尺寸折叠 |
| P2 | 次要字段、低频操作、辅助说明 | 宽屏展示，标准尺寸隐藏或进入更多菜单 |
| P3 | 装饰性内容、弱提示、非必要图形 | 可移除，不占用关键空间 |

### 2.4 横向空间给表格，纵向空间给流程

企业数据管理系统最重要的空间通常不是大图、卡片或营销式内容，而是表格、表单和详情。响应式布局应优先保证：

- 表格有足够横向空间。
- 表单字段不拥挤。
- 详情面板不会挤压主列表到不可读。
- 操作区不会占用过多纵向空间。

### 2.5 不依赖字体缩放解决布局问题

字体大小应由设计系统定义，不应随窗口宽度连续缩放。窗口变窄时应通过折叠、换行、滚动、隐藏低优先级内容、进入更多菜单等方式处理，而不是把文字无限缩小。

## 3. 尺寸断点体系

项目应建立统一断点，避免每个页面自行判断宽度。

建议断点如下：

| 模式 | 宽度范围 | 典型尺寸 | 设计目标 |
|---|---:|---|---|
| `compactDesktop` | `1280 <= width < 1440` | `1280 x 800` | 最小可用桌面模式，保主流程 |
| `standardDesktop` | `1440 <= width < 1920` | `1440 x 900`、`1600 x 900` | 标准桌面工作台 |
| `wideDesktop` | `1920 <= width < 2560` | `1920 x 1080` | 支持主列表 + 详情联动 |
| `ultraWideDesktop` | `width >= 2560` | `2560 x 1440`、`3440 x 1440` | 支持多栏信息增强，但限制单区最大宽度 |

高度也需要纳入判断：

| 高度范围 | 策略 |
|---|---|
| `height < 800` | 不作为正常支持范围，窗口最小高度应阻止进入该状态 |
| `800 <= height < 900` | 操作栏、筛选区、分页区必须紧凑，详情区域使用内部滚动 |
| `900 <= height < 1080` | 标准高度，适合常规工作台 |
| `height >= 1080` | 可展示更多行数、更多详情摘要和更多统计信息 |

建议在代码层定义类似结构：

```dart
class AppBreakpoints {
  static const compactDesktop = 1280.0;
  static const standardDesktop = 1440.0;
  static const wideDesktop = 1920.0;
  static const ultraWideDesktop = 2560.0;
}
```

响应式判断应集中封装，避免业务页面散落大量 `MediaQuery.of(context).size.width` 判断。

## 4. 窗口层策略

窗口层由 `window_manager` 或平台原生 runner 负责，重点解决应用外层边界。

### 4.1 默认窗口尺寸

推荐默认尺寸：

```text
1280 x 800
```

当前项目已经采用该默认尺寸。这个尺寸适合作为最小工作台验证基线。

如果后续业务页面复杂度提升，可以评估改为：

```text
1440 x 900
```

但是否调整默认尺寸应由真实业务页面验证决定，不应仅因为视觉上更宽松就提前修改。

### 4.2 最小窗口尺寸

推荐保持：

```text
minimumSize: 1280 x 800
```

原因：

- 低于 `1280` 宽度时，侧边栏、表格、筛选区和操作区很难同时可用。
- 企业桌面系统的核心任务不是窄屏浏览，而是高效处理业务数据。
- 设置最小尺寸比在极端窄窗口下做大量降级更稳定。

### 4.3 最大化与全屏

应用应自然支持最大化，但不建议强制全屏。最大化状态下应增加可用信息量，例如：

- 表格展示更多列。
- 详情面板常驻右侧。
- 筛选条件可展开更多字段。
- 统计摘要可从紧凑模式变为完整模式。

全屏适合看板、展示屏、监控页，不应作为日常业务页面的默认模式。

### 4.4 记住窗口尺寸

后续可以考虑保存用户上次窗口状态：

- 窗口位置。
- 窗口尺寸。
- 是否最大化。

建议使用 `getBounds()` 保存完整边界，而不是只保存 `getSize()`。

但第一阶段不必强制实现，避免在基础布局未稳定前增加状态恢复复杂度。

## 5. 应用主框架响应式策略

主框架通常由侧边导航、顶部栏、内容区和状态反馈区组成。

### 5.1 侧边导航

建议侧边导航分三种状态：

| 模式 | 宽度 | 适用断点 | 行为 |
|---|---:|---|---|
| 展开 | `220-240` | `standardDesktop` 及以上 | 展示图标与中文菜单名称 |
| 紧凑 | `64-80` | `compactDesktop` | 只展示图标，鼠标悬停显示提示 |
| 临时抽屉 | 按需 | 特殊页面或更复杂场景 | 不作为默认桌面策略 |

当前项目最小宽度为 `1280`，因此不建议把侧边栏做成移动端抽屉作为默认方案。紧凑侧边栏更适合桌面系统。

### 5.2 顶部栏

顶部栏应保持稳定高度，推荐范围：

```text
48 - 56
```

顶部栏内容优先级：

1. 当前模块标题。
2. 面包屑或当前上下文。
3. 全局搜索或快速入口。
4. 用户、环境、通知、设置。

当宽度不足时：

- 模块标题保留。
- 面包屑可截断。
- 次要入口进入更多菜单。
- 长文本使用 `TextOverflow.ellipsis`。

### 5.3 内容区

内容区应使用弹性布局，不应写死单一宽度。建议规则：

- 页面外层使用 `SizedBox.expand()` 或等价布局占满可用区域。
- 页面内部使用 `LayoutBuilder` 获取实际内容宽度。
- 主内容区允许内部滚动，但避免整个页面出现混乱的多层滚动。
- 不要让 `Row` 中的子组件无约束展开，必须使用 `Expanded`、`Flexible` 或固定宽度策略。

### 5.4 状态反馈区

加载、空数据、错误、权限不足等状态必须适配不同宽度。

建议：

- 空状态不要占据过高空间。
- 错误说明保持简洁，详细错误进入展开区。
- 主要重试按钮始终可见。
- 技术错误码可以展示，但用户说明使用中文。

## 6. 页面级布局策略

### 6.1 工作台页面

工作台页面建议采用 `Queue-first` 或 `Split-detail` 模式。

`compactDesktop`：

```text
左侧导航 + 主任务列表
详情通过抽屉、弹窗或页面内下钻打开
```

`standardDesktop`：

```text
左侧导航 + 主列表 + 可选摘要区
详情可以通过右侧抽屉打开
```

`wideDesktop`：

```text
左侧导航 + 主列表 + 常驻详情面板
```

`ultraWideDesktop`：

```text
左侧导航 + 主列表 + 详情面板 + 辅助信息区
```

注意：超宽屏下不应无限拉宽单列表。主表格可以设置合理最大宽度，额外空间用于详情、审计、附件、关联单据等辅助信息。

### 6.2 列表页面

列表页面通常包括：

- 页面标题。
- 筛选区。
- 操作栏。
- 数据表格。
- 分页区。

响应式策略：

| 区域 | `compactDesktop` | `standardDesktop` | `wideDesktop` |
|---|---|---|---|
| 标题 | 单行，长标题截断 | 单行完整展示 | 单行完整展示 |
| 筛选区 | 常用条件 1 行，更多折叠 | 2 行内展示 | 展示更多条件 |
| 操作栏 | 主按钮 + 更多菜单 | 常用按钮展示 | 更多按钮直接展示 |
| 表格 | 固定核心列，横向滚动 | 展示主要列 | 展示更多列 |
| 分页 | 紧凑分页 | 标准分页 | 标准分页 + 统计 |

### 6.3 详情页面

详情页面需要区分信息密度。

`compactDesktop`：

- 优先展示基础信息与当前状态。
- 审计记录、附件、关联单据使用标签页或折叠面板。
- 宽表格或明细列表使用内部横向滚动。

`standardDesktop`：

- 基础信息与业务详情可以左右分区。
- 操作区固定在顶部或右上角。

`wideDesktop`：

- 左侧主详情。
- 右侧状态流、审计记录、关联信息常驻。

### 6.4 表单页面

表单不应在宽屏下无限拉长字段。建议使用响应式列数：

| 模式 | 表单列数 | 字段宽度策略 |
|---|---:|---|
| `compactDesktop` | `1-2` | 关键字段优先，长字段独占一行 |
| `standardDesktop` | `2-3` | 常规字段分栏 |
| `wideDesktop` | `3-4` | 字段分组展示 |
| `ultraWideDesktop` | `3-4` | 限制表单最大宽度，避免过长输入框 |

表单按钮策略：

- 主按钮固定在操作区。
- 次按钮靠近主按钮。
- 危险操作需要明显区分。
- 长表单建议底部固定操作栏，但必须避免遮挡内容。

## 7. 业务组件响应式策略

### 7.1 数据表格

数据表格是本项目最重要的组件之一。

表格必须支持：

- 核心列固定。
- 列宽约束。
- 长文本截断。
- 单元格 Tooltip。
- 横向滚动。
- 分页或虚拟滚动。
- 批量选择。
- 行状态标识。

列优先级建议：

| 列类型 | 优先级 | 策略 |
|---|---|---|
| 业务编号 | P0 | 始终展示，可固定 |
| 状态 | P0 | 始终展示 |
| 客户 / 供应商 / 产品核心名称 | P0 | 始终展示，超长截断 |
| 金额 / 数量 / 日期 | P1 | 标准尺寸展示 |
| 备注 / 描述 / 附加字段 | P2 | 宽屏展示，窄屏进入详情 |
| 审计字段 | P2/P3 | 宽屏或详情中展示 |

表格宽度不足时，不应把所有列压得过窄，而应：

1. 固定 P0 列。
2. 隐藏或延后展示 P2/P3 列。
3. 保留横向滚动。
4. 将低优先级字段放入详情面板。

### 7.2 筛选区

筛选区应区分常用筛选和高级筛选。

常用筛选建议包括：

- 关键词。
- 状态。
- 日期范围。
- 业务类型。

高级筛选可以包括：

- 多级分类。
- 金额范围。
- 供应商属性。
- 仓库条件。
- 操作人。
- 审计状态。

响应式策略：

| 模式 | 策略 |
|---|---|
| `compactDesktop` | 常用筛选单行或两行，其他进入“更多筛选” |
| `standardDesktop` | 常用筛选完整展示，高级筛选折叠 |
| `wideDesktop` | 可展示更多筛选项 |
| `ultraWideDesktop` | 筛选项仍需分组，不要铺满整屏 |

筛选区应避免频繁改变表格位置。展开高级筛选时可以压缩表格高度，但不要导致操作栏跳动不可控。

### 7.3 操作栏

操作栏应有明确优先级。

| 操作类型 | 策略 |
|---|---|
| 主操作 | 始终展示，例如“新建订单” |
| 高频操作 | 标准尺寸展示，例如“导入”“导出” |
| 批量操作 | 选择数据后展示 |
| 低频操作 | 进入更多菜单 |
| 危险操作 | 独立确认，不与普通操作混放 |

当宽度不足时，保留主操作和当前上下文相关操作，其余进入更多菜单。

### 7.4 详情面板

详情面板适合展示当前选中记录的摘要、状态流、操作记录、附件和关联单据。

建议宽度：

| 模式 | 详情面板策略 |
|---|---|
| `compactDesktop` | 默认不常驻，使用抽屉或弹窗 |
| `standardDesktop` | 可选右侧抽屉 |
| `wideDesktop` | 常驻右侧面板，宽度 `360-480` |
| `ultraWideDesktop` | 常驻右侧面板，必要时增加辅助信息区 |

详情面板不能把主表格压缩到不可读。若空间不足，应优先隐藏详情面板，而不是压缩主表格。

### 7.5 弹窗与抽屉

弹窗适合短流程，抽屉适合上下文详情和编辑。

弹窗宽度建议：

| 场景 | 推荐宽度 |
|---|---:|
| 确认操作 | `360-480` |
| 简单表单 | `520-720` |
| 复杂表单 | `800-960` |
| 大型编辑 | 优先使用页面或抽屉 |

抽屉宽度建议：

| 模式 | 推荐宽度 |
|---|---:|
| `compactDesktop` | `640-720` |
| `standardDesktop` | `720-860` |
| `wideDesktop` | `860-1040` |

弹窗与抽屉必须支持内部滚动，底部操作按钮应保持稳定可见。

### 7.6 卡片与统计区

统计卡片只作为辅助信息，不应挤占表格和操作空间。

建议：

- `compactDesktop` 下最多展示 2-4 个核心指标。
- `standardDesktop` 下展示一行核心指标。
- `wideDesktop` 下可以展示趋势、异常摘要或分组统计。
- 不要为了填满宽屏而增加无业务价值的卡片。

## 8. 滚动策略

滚动必须可预测，避免页面出现多个方向混乱滚动。

建议规则：

- 页面整体尽量不横向滚动。
- 表格区域可以横向滚动。
- 长表单可以纵向滚动。
- 详情面板内部可以纵向滚动。
- 弹窗和抽屉内部可以纵向滚动。
- 顶部栏、侧边栏、操作栏不应被页面滚动带走，除非页面设计明确需要。

对于主列表页面，推荐结构：

```text
Page
  Header fixed
  Filter area fixed or collapsible
  Toolbar fixed
  Table flexible scroll area
  Pagination fixed
```

这样可以保证用户在浏览大量数据时，筛选、操作和分页始终可访问。

## 9. 文本与密度策略

### 9.1 字体

字体大小建议：

| 场景 | 建议字号 |
|---|---:|
| 页面标题 | `18-22` |
| 区块标题 | `16-18` |
| 表格正文 | `13-14` |
| 表单标签 | `13-14` |
| 辅助说明 | `12-13` |

不建议根据窗口宽度连续缩放字体。不同密度模式可以使用有限档位，而不是无级缩放。

### 9.2 文本溢出

所有可能变长的文本都必须有溢出策略：

- 菜单名称：截断 + Tooltip。
- 表格单元格：截断 + Tooltip。
- 页面标题：必要时截断。
- 按钮文本：避免过长，低频操作进入更多菜单。
- 详情字段：允许换行，但限制最大高度。

### 9.3 间距与密度

建议建立密度模式：

| 模式 | 适用场景 | 特点 |
|---|---|---|
| `compactDensity` | `1280 x 800` | 间距较小，控件高度紧凑 |
| `standardDensity` | `1440 x 900` | 默认密度 |
| `comfortableDensity` | 宽屏或低密度页面 | 间距更舒展 |

表格和表单应以 `compactDensity`、`standardDensity` 为主，不建议默认使用过大的卡片式间距。

## 10. 实现架构建议

响应式能力应放入 `shared/` 或 `app/` 的基础层，避免分散在业务页面里。

建议目录：

```text
frontend/
  lib/
    app/
      layout/
        app_breakpoints.dart
        app_layout_mode.dart
        app_responsive_scope.dart
    shared/
      widgets/
        responsive/
          responsive_builder.dart
          responsive_grid.dart
          responsive_page.dart
          responsive_toolbar.dart
          responsive_filter_bar.dart
```

### 10.1 `AppLayoutMode`

建议定义统一布局模式：

```dart
enum AppLayoutMode {
  compactDesktop,
  standardDesktop,
  wideDesktop,
  ultraWideDesktop,
}
```

业务页面只关心当前 `AppLayoutMode`，不直接关心具体宽度数字。

### 10.2 `ResponsiveBuilder`

建议封装一个响应式构建器：

```dart
ResponsiveBuilder(
  builder: (context, layoutMode, constraints) {
    return const SizedBox.expand();
  },
)
```

用途：

- 统一读取 `LayoutBuilder` 结果。
- 统一计算当前断点。
- 避免每个页面重复写判断逻辑。

### 10.3 `ResponsivePage`

建议封装页面骨架：

- 页面标题。
- 筛选区。
- 操作栏。
- 主内容区。
- 详情区。
- 分页区。

业务页面只提供内容，不重复处理主框架布局。

### 10.4 `ResponsiveGrid`

用于表单和统计区。根据 `layoutMode` 自动决定列数、间距和字段跨列规则。

### 10.5 `ResponsiveToolbar`

统一处理操作按钮优先级：

- P0 操作始终展示。
- P1 操作标准尺寸展示。
- P2 操作进入更多菜单。
- 批量操作根据选择状态展示。

## 11. GetX 配合方式

`GetX` 不应承载布局计算本身。推荐边界如下：

| 层级 | 职责 |
|---|---|
| `Widget` | 根据 `layoutMode` 组合 UI |
| `GetxController` | 管理业务状态、筛选条件、加载状态、选择状态 |
| `Service` / `Repository` | 调用 API 与处理数据 |
| `ResponsiveBuilder` | 计算布局模式 |

不要把窗口宽度、断点判断、组件列数等逻辑写进业务 `Controller`。这些属于视图层行为。

## 12. 不同业务模块适配建议

本章内容只作为响应式设计时的业务场景参考，用于说明不同页面类型可能如何使用断点、表格、详情面板、筛选区和操作栏。

本章不作为后续业务页面实现的必须准则。后续订单、供应商、产品、仓储、发运等业务页面的具体信息架构、交互模式、字段优先级和页面形态，应在对应业务模块启动设计或开发时重新确认。

### 12.1 订单模块

订单模块可以参考主列表 + 详情联动的方式，但最终形态以后续订单业务页面设计为准。

- `compactDesktop`：订单列表为主，详情通过抽屉打开。
- `standardDesktop`：列表为主，可打开右侧详情抽屉。
- `wideDesktop`：列表 + 右侧订单详情常驻。
- `ultraWideDesktop`：列表 + 详情 + 状态流 / 关联单据。

订单列表 P0 列参考示例：

- 订单编号。
- 状态。
- 客户 / 供应商。
- 金额。
- 交付日期。
- 异常标识。

### 12.2 供应商模块

供应商模块可以关注资料完整度、状态、关联产品和交易记录，但具体页面结构以后续供应商模块设计为准。

- 小尺寸优先展示供应商主数据。
- 宽屏展示关联产品、最近订单、风险提示。
- 详情信息可分为基础信息、联系人、结算信息、审计记录。

### 12.3 产品模块

产品模块可能需要兼顾图片、规格、库存、价格和供应关系，但具体字段呈现方式以后续产品模块设计为准。

- 表格中图片缩略图可限制尺寸。
- 规格字段容易过长，可截断并在详情中完整展示。
- 宽屏可展示库存、价格、供应商关系。

### 12.4 仓储与发运模块

仓储和发运模块通常会涉及状态流和批量操作，但具体流程、页面结构和操作入口以后续业务设计为准。

- 状态列、异常列、时间列优先级高。
- 宽屏下可展示路线、发运节点、签收状态。
- 批量操作栏可保持稳定，避免因窗口变化影响高频处理效率。

## 13. 验证标准

### 13.1 必测窗口尺寸

每个核心页面至少验证以下尺寸：

| 尺寸 | 目的 |
|---|---|
| `1280 x 800` | 最小可用尺寸 |
| `1440 x 900` | 标准桌面尺寸 |
| `1600 x 900` | 常见办公显示器 |
| `1920 x 1080` | 宽屏标准尺寸 |
| `2560 x 1440` | 高分屏 / 大屏 |
| `3440 x 1440` | 超宽屏 |

### 13.2 必测行为

每个核心页面至少验证：

- 页面无 `overflow`。
- 侧边导航可用。
- 顶部栏文字不重叠。
- 筛选区可展开与收起。
- 表格核心列可见。
- 表格横向滚动可用。
- 主操作按钮可见。
- 批量操作状态正确。
- 详情面板不会压坏主表格。
- 弹窗和抽屉在最小尺寸下可完整操作。

如果在 `Windows` 环境验证：

### 13.4 Widget Test 建议

后续实现响应式组件时，应增加固定尺寸的 widget test。测试重点不是像素级截图，而是验证不同尺寸下的关键组件是否存在、是否隐藏、是否进入更多菜单。

示例测试场景：

- `1280 x 800` 下侧边栏为紧凑模式。
- `1440 x 900` 下侧边栏为展开模式。
- `1920 x 1080` 下详情面板常驻。
- 操作按钮超过宽度时，低优先级按钮进入更多菜单。
- 长文本不会导致 overflow。

## 14. 分阶段实施建议

### 阶段一：基础能力

目标：建立响应式基础设施。

任务：

- 定义 `AppBreakpoints`。
- 定义 `AppLayoutMode`。
- 实现 `ResponsiveBuilder`。
- 实现基础 `ResponsivePage`。
- 建立最小尺寸下的布局测试。

### 阶段二：主框架适配

目标：让应用壳在不同尺寸下稳定。

任务：

- 侧边导航支持展开与紧凑。
- 顶部栏支持长文本截断。
- 内容区使用弹性布局。
- 操作入口进入更多菜单。

### 阶段三：业务组件适配

目标：让表格、筛选、表单、详情具备统一响应式行为。

任务：

- 表格列优先级。
- 筛选区折叠。
- 表单响应式列数。
- 详情面板宽度策略。
- 弹窗与抽屉尺寸策略。

### 阶段四：核心业务页面验证

目标：选择一个后续已确认设计的核心业务页面，作为第一条完整响应式验证样板线。

任务：

- 列表区域。
- 详情区域。
- 筛选区域。
- 批量操作。
- 异常或关键状态处理。

样板页面验证通过后，只复用其中的响应式基础能力和验证方法，不直接复制业务页面的信息架构或交互形态。

## 16. 完成定义

当响应式能力实现后，应满足以下标准：

- `1280 x 800` 下核心业务可完成。
- `1440 x 900` 下达到标准桌面工作台体验。
- `1920 x 1080` 下可以自然展示详情联动。
- 超宽屏下不会出现内容被无限拉宽的问题。
- 表格、筛选、表单、详情、弹窗均有明确降级策略。
- 自动化测试覆盖关键断点。
- 真实桌面运行验证无明显布局溢出。

该方案中的断点、窗口边界、布局稳定性、滚动策略和验证标准，是后续响应式实现的项目级约束。业务模块适配建议只作为参考示例，不限定后续业务页面的具体设计；后续新增业务页面时，应结合当时的业务需求重新确认页面结构和交互方案。
