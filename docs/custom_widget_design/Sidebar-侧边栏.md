# Sidebar 侧边栏组件设计方案

> 适用项目：`QSDMS_DESKTOP`
> 适用端：`Flutter Desktop`，目标平台为 `macOS`、`Windows`
> 技术栈：`Flutter Desktop + GetX + window_manager`
> 文档目标：定义桌面端主框架侧边栏组件的结构、视觉、交互、状态、数据合同与验证标准，作为后续实现 `QsdmsSidebar` 的设计依据。

## 1. 设计结论

`Sidebar` 是 QSDMS 桌面端主框架的左侧导航组件，负责承载品牌识别、模块导航、公告入口和当前登录用户入口。

本项目定位是企业数据管理系统，核心用户是内部运营人员，主要任务是处理企业运营中的高频业务数据。因此侧边栏不应照搬参考图中偏营销 SaaS 的大面积装饰卡片风格，而应采用低干扰、高识别、可长时间工作的企业工作台风格。

设计结论如下：

| 项目 | 结论 |
|---|---|
| 默认形态 | 左侧固定宽度导航栏 |
| 展开宽度 | `240`，最长不超过 `260` |
| 折叠宽度 | `72` |
| 背景 | `#FFFFFF` |
| 页面背景承接 | `#F6F8FA`，与当前 `AppTheme` 保持一致 |
| 主色 | `#1F6FEB`，沿用当前 `ColorScheme.fromSeed` 的项目主色 |
| 选中背景 | `#EAF2FF` |
| 边界线 | 右侧 `1px` 视觉分隔，建议 `#E5EAF0` |
| 菜单项高度 | 展开 `44-48`，折叠 `44-48` |
| 菜单图标 | `20-22` |
| 字体密度 | 企业桌面密度，不使用营销页大字号 |
| 折叠策略 | 只保留图标、激活态、徽标提示和用户头像，隐藏品牌文字与公告卡片 |
| 桌面断点 | `1280 x 800` 使用折叠优先，`1440 x 900` 及以上使用展开优先 |

宽度没有采用参考值 `260-280` 作为默认值，原因是当前项目最小窗口宽度为 `1280`。侧边栏过宽会挤压表格、筛选区和详情面板，违背企业数据系统“横向空间优先给表格”的原则。`240` 可以容纳绝大多数中文菜单名称，同时保留主内容区可用空间。

## 2. 真相源

本方案基于以下项目事实：

- `docs/技术栈规范文档.md`：项目定位为 `Flutter Desktop + GetX + NestJS + PostgreSQL` 的企业数据管理系统。
- `docs/响应式设计方案.md`：当前最小窗口为 `1280 x 800`，侧边导航在 `compactDesktop` 下应使用紧凑模式，在 `standardDesktop` 及以上应使用展开模式。
- `frontend/lib/app/theme/app_theme.dart`：当前浅色主题基线为 `scaffoldBackgroundColor: #F6F8FA`，主色来自 `seedColor: #1F6FEB`。
- `frontend/lib/main.dart`：窗口默认尺寸与最小尺寸均为 `1280 x 800`。
- `frontend/lib/modules/home/home_page.dart`：当前首页仍是空白骨架，适合先定义主框架组件设计再实现。
- 用户提供的参考图：用于参考结构关系和状态表达，不作为本项目视觉风格的唯一依据。

## 3. 组件职责

`Sidebar` 只负责主框架导航与账户入口，不承载业务页面逻辑。

必须负责：

- 展示品牌区。
- 展示菜单列表。
- 标识当前菜单选中状态。
- 展示图标、文案和可选徽标。
- 支持展开与折叠两种状态。
- 支持菜单区域内部滚动。
- 展示底部通知公告卡片。
- 展示当前用户基础信息。
- 点击用户区后展示个人信息与退出登录入口。

不应负责：

- 直接请求业务列表数据。
- 判断复杂业务权限规则。
- 直接拼接 API 路径。
- 保存登录态。
- 打开业务弹窗。
- 实现页面内容区布局。
- 在组件内部硬编码真实公告链接、用户隐私信息或环境配置。

## 5. 布局规则

### 5.1 外层尺寸

| 状态 | 宽度 | 说明 |
|---|---:|---|
| 展开 | `240` | 默认展开宽度，适合中文菜单 |
| 展开上限 | `260` | 菜单名称普遍较长时可调整到上限 |
| 折叠 | `72` | 只展示图标与必要状态 |

高度规则：

- 使用父级主框架给定的完整高度。
- 侧边栏自身保持全高。
- 菜单区使用剩余高度并允许纵向滚动。
- 品牌区、公告区、用户区不随菜单列表滚动。

不建议：

- 不使用移动端抽屉作为桌面端默认策略。
- 不让侧边栏随页面内容滚动。
- 不在 `1280 x 800` 下默认使用 `280` 宽度。

### 5.2 展开状态间距

| 区域 | 建议值 |
|---|---:|
| 外层水平内边距 | `12-16` |
| 品牌区高度 | `64` |
| 品牌 Logo 尺寸 | `32` |
| 菜单项高度 | `44-48` |
| 菜单项圆角 | `8` |
| 菜单项水平内边距 | `12` |
| 图标尺寸 | `20-22` |
| 图标与文字间距 | `12` |
| 徽标最小宽度 | `18` |
| 用户区高度 | `64-72` |

菜单项高度不采用 `56-64` 作为默认值。这个高度适合偏展示型后台，但 QSDMS 是高频数据处理工具，侧边栏需要容纳更多业务模块，默认应更紧凑。

### 5.3 折叠状态间距

| 区域 | 建议值 |
|---|---:|
| 外层宽度 | `72` |
| 菜单项容器 | `44 x 44` |
| 图标尺寸 | `20-22` |
| 激活指示 | 左侧 `3` 宽竖条或圆角背景 |
| 徽标 | 右上角小圆点或短数字 |
| 用户头像 | `36` |

折叠状态必须提供 `Tooltip`，内容为完整中文菜单名称。对于带徽标的菜单，`Tooltip` 应包含徽标含义，例如 `消息，12 条未读`。

## 6. 视觉规范

### 6.1 颜色

| 用途 | 推荐色值 | 说明 |
|---|---|---|
| 侧边栏背景 | `#FFFFFF` | 保持清爽，不与内容区抢层级 |
| 应用背景 | `#F6F8FA` | 沿用当前主题 |
| 主色 | `#1F6FEB` | 与当前 `AppTheme` 一致 |
| 激活背景 | `#EAF2FF` | 浅蓝，不刺眼 |
| 激活文字 | `#1F6FEB` | 强识别 |
| 默认文字 | `#344054` | 适合长时间阅读 |
| 次级文字 | `#667085` | 用户角色、说明文案 |
| 默认图标 | `#667085` | 降低噪音 |
| 分隔线 | `#E5EAF0` | 轻边界 |
| 危险操作 | `#D92D20` | 退出登录等危险操作 |
| 徽标红色 | `#D92D20` | 未读、异常 |
| 徽标蓝色 | `#1F6FEB` | 普通数量 |
| 徽标橙色 | `#B54708` | 待处理、预警 |

不建议继续使用参考图中的高饱和蓝紫 `#4f46ff` 作为默认主色，因为当前项目主题已经稳定为 `#1F6FEB`，且企业桌面系统更需要克制、清晰、稳定的色彩体系。

### 6.2 字体层级

| 场景 | 字号 | 字重 |
|---|---:|---:|
| 品牌名称 | `16` | `600` |
| 菜单文字 | `14` | `500` |
| 菜单激活文字 | `14` | `600` |
| 菜单分组标题 | `12` | `600` |
| 徽标数字 | `11-12` | `600` |
| 用户名 | `13-14` | `600` |
| 用户角色 | `12` | `400` |

所有长文本都必须定义溢出策略：

- 品牌名称：单行截断。
- 菜单名称：单行截断，悬停显示 `Tooltip`。
- 用户名：单行截断。
- 用户角色：单行截断。
- 公告标题：最多两行，超出截断。

### 6.3 图标

当前 `pubspec.yaml` 已启用 `uses-material-design: true`，且依赖中有 `cupertino_icons`。第一阶段不新增图标依赖，优先使用 Flutter 自带的 `Icons` 或 `CupertinoIcons`。

图标规则：

- 每个一级菜单必须有图标。
- 图标语义必须和业务模块匹配。
- 激活状态下图标颜色跟随激活文字颜色。
- 默认状态下图标颜色为次级文字色。
- 禁用状态下降低透明度，不只依靠颜色表达。
- 折叠状态下图标是主要识别符，必须保持稳定。

后续如果现有图标无法满足业务模块表达，再单独评估是否引入图标库。新增第三方依赖前必须先检查 `pubspec.yaml` 并确认引入必要性。

## 7. 状态设计

### 7.1 侧边栏整体状态

```dart
enum SidebarDisplayMode {
  expanded,
  collapsed,
}
```

状态来源建议：

- `compactDesktop`：默认 `collapsed`。
- `standardDesktop` 及以上：默认 `expanded`。
- 用户手动切换：第一阶段只在当前运行会话内生效。
- 状态持久化：等主框架稳定后再做，不作为首版必要能力。

### 7.2 菜单项状态

| 状态 | 视觉 |
|---|---|
| 默认 | 白底，灰色图标，深灰文字 |
| 悬停 | 浅灰或浅蓝背景，图标略加深 |
| 按下 | 背景加深一档，内容轻微压低 |
| 激活 | 浅蓝背景，主色图标与文字，左侧可增加 `3px` 指示条 |
| 禁用 | 透明度降低，禁用点击，保留 Tooltip 说明 |
| 聚焦 | 显示键盘可见焦点框 |

激活状态必须由当前路由或当前菜单 `id` 驱动，不能只靠本地点击状态。刷新路由、深链跳转或程序化导航后，侧边栏高亮必须保持正确。

### 7.3 徽标状态

徽标类型：

| 类型 | 用途 | 展开状态 | 折叠状态 |
|---|---|---|---|
| `count` | 未读数量、待处理数量 | 显示数字 | 显示小圆点或短数字 |
| `dot` | 有更新、有提醒 | 显示圆点 | 显示圆点 |
| `label` | `new`、`beta` 等短标签 | 显示短文本 | 显示圆点，Tooltip 展示完整含义 |
| `severity` | 异常等级 | 使用颜色表达等级 | 使用颜色圆点 |

数量规则：

- `0` 不展示徽标。
- `1-99` 展示真实数字。
- `>99` 展示 `99+`。
- 业务异常徽标优先级高于普通未读徽标。

## 8. 菜单数据合同

菜单数据应由配置或路由层提供，`Sidebar` 只消费结构化数据。

建议模型：

```dart
class SidebarMenuItemConfig {
  /// 菜单唯一标识，保持英文，不能使用展示文案作为主键。
  final String id;

  /// 用户可见中文菜单名称。
  final String label;

  /// GetX 路由名称，例如 AppRoutes.home。
  final String routeName;

  /// Flutter 图标数据，第一阶段优先使用 Icons 或 CupertinoIcons。
  final IconData icon;

  /// 菜单徽标配置，没有徽标时为空。
  final SidebarBadgeConfig? badge;

  /// 权限编码由外层过滤后传入，Sidebar 不直接计算复杂权限。
  final String? permissionCode;

  /// 是否可点击。权限不足、功能未开放或系统维护时可禁用。
  final bool enabled;

  /// 子菜单。首版最多支持两级，不做无限递归菜单。
  final List<SidebarMenuItemConfig> children;
}
```

首版建议只实现一级菜单，或者实现一级菜单加轻量二级菜单。不要在首版做无限层级菜单，原因是企业桌面导航应保持可扫读，复杂层级会降低效率。

### 8.1 推荐初始菜单

推荐初始菜单只保留最高频、最稳定的主入口，避免在主框架阶段提前铺开过多业务模块。后续具体业务模块可挂在 `基础数据` 或业务页面内部，再根据真实业务边界决定是否提升为一级菜单。

| 菜单 | 技术 id | 图标建议 | 说明 |
|---|---|---|---|
| 工作台 | `dashboard` | `Icons.dashboard_outlined` | 总览与待办 |
| 基础数据 | `baseData` | `Icons.dataset_outlined` | 主数据、基础档案、公共配置 |
| 系统设置 | `settings` | `Icons.settings_outlined` | 账号、权限、基础配置 |

如果实际路由未实现，菜单项应显示为禁用或不显示，不应点击后进入空页面。

## 9. 品牌区设计

品牌区目标是明确当前系统身份，不做过度装饰。

展开状态：

- 左侧展示品牌 Logo。
- 右侧展示 `QSDMS` 或 `千树数据管理系统`。
- 可展示简短环境标识，例如 `DEV`、`TEST`，但生产环境不建议显眼显示。
- 品牌区可包含折叠切换按钮，位置建议在右侧。

折叠状态：

- 只展示 Logo。
- 折叠切换按钮可在品牌区下方或悬停时出现。
- 品牌完整名称通过 `Tooltip` 提供。

品牌区不建议作为返回首页的唯一入口。点击 Logo 可以导航到工作台，但菜单中仍应保留“工作台”入口。

## 10. 菜单区设计

菜单区是侧边栏的核心区域。

布局要求：

- 使用可滚动区域承载菜单列表。
- 菜单列表上下保留 `8-12` 的呼吸空间。
- 一级菜单之间不使用厚重分割线。
- 若需要分组，使用小号分组标题，而不是卡片分组。
- 滚动条默认保持低存在感，悬停或滚动时增强可见性。

菜单点击行为：

- 可用菜单点击后触发 `Get.toNamed(routeName)` 或主框架传入的 `onMenuSelected`。
- 当前已激活菜单再次点击不重复导航。
- 禁用菜单点击无动作，并显示禁用原因。
- 菜单跳转失败时由路由层或页面层处理错误，不在 `Sidebar` 内吞掉异常。

键盘行为：

- `Tab` 可以进入侧边栏。
- `ArrowUp` / `ArrowDown` 可以在菜单间移动焦点。
- `Enter` 或 `Space` 触发当前聚焦菜单。
- `Esc` 可关闭用户弹窗或菜单浮层。

## 11. 通知公告卡片设计

参考图中的 `Upgrade Card` 更像订阅升级入口。QSDMS 是内部企业系统，默认更适合设计为“通知公告卡片”或“系统更新卡片”。只有项目确实存在版本升级、桌面客户端更新或服务公告时，才使用升级语义。

展开状态显示：

- 卡片位于底部用户区上方。
- 卡片高度建议 `96-128`。
- 卡片圆角 `8`。
- 背景可使用浅蓝或浅灰，不使用大面积高饱和渐变。
- 可包含一张本地图片或远程图片。
- 图片和卡片整体可点击，点击后打开配置的网站链接。

折叠状态隐藏：

- 折叠后不展示公告卡片，避免占用窄侧栏空间。
- 如有重要公告，可在通知菜单上使用徽标提示。

链接规则：

- 链接 URL 不应硬编码在组件内部。
- URL 应由配置或接口下发，并经过域名白名单校验。
- 打开外部链接建议由外层注入 `onNoticeTap` 回调。
- 如果后续要直接在 Flutter 中打开浏览器，需要评估引入 `url_launcher`。当前依赖中没有该包，不能在实现时直接假设可用。

图片规则：

- 优先使用本地资源作为默认图，避免网络不可用导致底部区域空白。
- 远程图片必须有加载失败占位。
- 图片不展示敏感业务数据。
- 图片尺寸必须稳定，不能因为加载完成导致侧边栏高度跳动。

## 12. 用户信息区设计

用户信息区固定在侧边栏底部，承载当前登录用户身份和账户操作入口。

### 12.1 视觉方向

用户信息区采用“白色账号胶囊”方案。目标是在不抢占菜单高亮层级的前提下，让底部账号入口更轻快、更像一个独立的账户卡片。

展开状态的用户区使用白色背景、较大圆角、细边框和轻量阴影。头像使用温和粉色圆角底承载头像芯，形成参考图中的活泼感；右侧使用低强调退出 / 账户操作图标，只作为账户入口提示。

折叠状态仍只展示头像，不展示姓名、角色和右侧操作图标。折叠态必须保留 `Tooltip`，内容包含用户名和角色，避免紧凑侧边栏丢失身份信息。

本阶段不扩展用户数据模型，不新增真实在线状态接口。右侧图标不直接触发退出登录，真正退出登录继续通过账户弹窗动作完成。

展开状态：

```text
Avatar  用户名              ActionIcon
        角色 / 组织
```

折叠状态：

```text
Avatar
```

展示内容：

| 字段 | 展开状态 | 折叠状态 |
|---|---|---|
| 头像 | 展示 | 展示 |
| 用户名 | 展示，超出截断 | Tooltip 中展示 |
| 角色 | 展示，超出截断 | Tooltip 中展示 |
| 组织 | 可放入弹窗 | Tooltip 或弹窗中展示 |
| 在线状态 | 本阶段不展示 | 本阶段不展示 |

点击行为：

- 点击整个用户区打开 `SidebarUserPopover`。
- 弹窗应锚定在用户区右上方或右侧。
- 弹窗不应遮挡用户区本身。
- 点击外部区域关闭弹窗。
- 按 `Esc` 关闭弹窗。

弹窗内容：

```text
用户头像    用户名 / 角色                         Close

账号        当前账号
组织        当前组织
登录        最近登录时间

[修改个人信息]    [退出登录]
```

弹窗视觉：

- 弹窗背景使用白色。
- 弹窗内容必须有明确内边距，建议 `20` 左右。
- 信息区使用紧凑分组，行距保持可读但不做大留白。
- 底部两个操作按钮使用实心按钮，按钮之间保留中等间距。
- 右上角关闭按钮用于关闭弹窗，不触发账户动作。

退出登录规则：

- 当前阶段两个底部按钮只打印日志，不接真实接口。
- 后续接入真实退出时，`Sidebar` 只触发外层注入的退出流程。
- 真正退出登录由认证模块处理，并必须有确认，防止误点。
- 退出前不在日志中输出 Token、密码或隐私数据。

## 13. 展开与折叠策略

### 13.1 自动策略

侧边栏默认状态由 `AppLayoutMode` 决定：

| 模式 | 宽度范围 | 默认侧边栏 |
|---|---|---|
| `compactDesktop` | `1280 <= width < 1440` | 折叠 |
| `standardDesktop` | `1440 <= width < 1920` | 展开 |
| `wideDesktop` | `1920 <= width < 2560` | 展开 |
| `ultraWideDesktop` | `width >= 2560` | 展开 |

### 13.2 手动策略

用户可以手动切换展开与折叠：

- `compactDesktop` 下允许临时展开，但内容区必须保持可用。
- 手动切换使用平滑宽度动画。
- 动画只改变宽度、透明度和位移，不做复杂视觉特效。
- 首版不强制保存用户偏好，避免基础布局未稳定时增加状态复杂度。

### 13.3 折叠隐藏规则

折叠后隐藏：

- 品牌文字。
- 菜单文字。
- 菜单分组标题。
- 公告卡片。
- 用户名与角色。
- 标签型徽标文字。

折叠后保留：

- Logo。
- 菜单图标。
- 当前菜单激活态。
- 数字徽标或圆点。
- 用户头像。
- 必要 Tooltip。

## 15. 与 GetX 的关系

推荐边界：

| 层级 | 职责 |
|---|---|
| `GetX Route` | 提供当前路由状态 |
| `AppShell` | 根据路由与布局模式确定侧边栏状态 |
| `Sidebar` | 根据传入数据渲染 UI |
| `AuthController` | 提供当前用户信息和退出登录流程 |
| `PermissionController` | 在菜单传入前过滤权限 |

不建议：

- 不在 `Sidebar` 中直接读取多个业务 `Controller`。
- 不在 `Sidebar` 中直接做权限树计算。
- 不在 `Sidebar` 中直接写登录退出细节。
- 不把所有菜单配置散落在不同页面。

## 16. 可访问性与可用性

必须支持：

- 折叠状态下所有图标菜单都有 `Tooltip`。
- 菜单项有明确 `Semantics` 标签。
- 当前激活菜单不能只依赖颜色表达，还需要背景、指示条或语义状态。
- 键盘焦点可见。
- 禁用菜单有原因说明。
- 弹窗可以通过键盘关闭。
- 徽标信息在 `Tooltip` 或语义标签中可读。

鼠标体验：

- 菜单项悬停反馈清晰但克制。
- 用户区可点击范围覆盖整行。
- 外部链接点击前不出现复杂中断，但链接打开失败要有反馈。
- 危险操作使用确认弹窗，不和普通操作混放。

## 17. 动效规范

侧边栏动效用于帮助状态变化，不用于制造视觉噪音。

推荐：

- 展开 / 折叠宽度动画：`160-220ms`。
- 菜单 hover：`100-140ms`。
- 激活态切换：`120-180ms`。
- 用户弹窗出现：`120-160ms`，使用轻微透明度和位移。

不建议：

- 不做持续循环动画。
- 不做发光、霓虹、液态等装饰效果。
- 不让菜单图标在悬停时大幅移动。
- 不使用会影响低端 Windows 设备性能的复杂阴影或滤镜。

## 18. 错误与空状态

侧边栏虽然是框架组件，也需要处理异常输入。

| 场景 | 策略 |
|---|---|
| 菜单为空 | 展示最小空状态文案，保留品牌区和用户区 |
| 当前路由无匹配菜单 | 不高亮菜单，或高亮最近父级菜单 |
| 用户信息为空 | 展示默认头像和 `未登录`，禁用个人信息入口 |
| 公告图片加载失败 | 展示稳定占位，不改变卡片尺寸 |
| 公告链接为空 | 卡片不可点击，展示普通公告 |
| 菜单权限不足 | 外层过滤；如必须展示则禁用并说明原因 |

组件不应因为缺少可选字段而抛出运行时异常。

- 明确展开与折叠两种状态。
- 明确当前菜单高亮规则。
- 明确图标、文案、徽标的展示规则。
- 明确项目主色、背景、宽度、密度与参考图之间的取舍。

- 明确公告图片链接和用户弹窗的实现边界。
- 明确后续测试与真实桌面验证方式。

实现验收：

- `1280 x 800` 下主内容区仍可用于表格类页面。
- `1440 x 900` 下侧边栏展开且不显拥挤。
- 折叠状态可通过图标和 Tooltip 完成导航识别。
- 展开 / 折叠切换没有明显布局跳动。
- 菜单多于可视高度时，中间菜单区可滚动，底部用户区固定。

- 用户信息弹窗可通过鼠标和键盘关闭。
- 退出登录不在侧边栏内部直接处理认证细节。
- 无日志敏感信息泄露。

## 22. 最终组件定位

`Sidebar` 不是装饰性组件，而是 QSDMS 桌面端工作台的主导航基础设施。它的首要目标是让内部运营人员在高密度业务页面之间稳定、快速、低干扰地切换。

参考图提供了结构方向：品牌、菜单、公告卡片、用户入口和高亮状态。本项目需要保留这些结构优点，但在视觉上回到企业桌面系统所需的克制密度、稳定断点和可测试组件边界。
