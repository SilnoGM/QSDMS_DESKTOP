# 用户权限体系设计

> 文档位置：`docs/backend/user/用户权限体系设计.md`
> 适用范围：`QSDMS_DESKTOP` 单租户后台权限体系
> 当前结论：采用单租户 `RBAC`，不做多租户设计。

## 1. 设计目标

当前系统是 `Flutter Desktop + GetX` 前端和 `NestJS + PostgreSQL + Prisma` 后端的桌面优先企业数据管理系统。权限体系的目标是建立一套可以支撑登录、菜单、按钮、接口和后续审计的基础能力，同时避免过早引入复杂组织权限和多租户隔离。

本设计覆盖以下内容：

- 用户账号与用户资料分离。
- 一个用户可以绑定多个角色。
- 一个角色可以绑定多个权限。
- 权限分为菜单权限、操作权限、接口权限三类。
- 前端根据权限做菜单和按钮显隐。
- 后端通过 `Guard` 和权限服务做接口强校验。
- 登录后下发最小可用的用户信息、角色和权限集合。
- 所有公共 API、DTO、数据库 schema 变更进入实现阶段前需要再次确认。

不做以下内容：

- 不做多租户，不设计 `Tenant`、`tenantId`、租户成员、租户管理员。
- 不做每个用户单独配置权限的复杂模型，首版只通过角色授权。
- 不做行级数据权限和字段级权限，后续业务需要时再单独设计。
- 不做无限层级菜单，前端侧边栏首版最多两级菜单。
- 不把前端权限显隐当作安全边界，安全边界始终在后端。

## 2. 当前项目现状

只读追踪得到的当前状态如下：

| 范围 | 当前状态 | 对本设计的影响 |
|---|---|---|
| 后端框架 | `backend/` 已有 `NestJS`、`ConfigModule`、`PrismaModule`、`OpenAPI`、全局 `/api` 前缀和 `ValidationPipe` | 权限模块可以按标准 NestJS 模块接入 |
| Prisma schema | `backend/prisma/schema.prisma` 目前只有 `generator` 和 `datasource` | RBAC 表结构属于新增 schema，实施前必须再次确认 |
| 响应结构 | `ApiResponse<T>` 目前包含 `code`、`message`、`data` | 登录、用户信息、菜单接口应沿用该结构 |
| OpenAPI | `backend/src/openapi.ts` 已配置 `BearerAuth` | 鉴权接口和权限接口应进入 OpenAPI 契约 |
| 前端 HTTP | `frontend/lib/shared/services/api_client.dart` 已有统一 `Dio` 入口 | 后续可在这里统一注入 `Authorization` header |
| 前端路由 | `AppRoutes` 已有 `/`、`/base-data`、`/settings` | 菜单权限编码应与路由保持稳定映射 |
| 前端菜单 | `SidebarMenuItemConfig` 已预留 `permissionCode` | 可直接作为前端菜单过滤和禁用依据 |
| 侧边栏文档 | 现有设计要求 `Sidebar` 只消费结构化菜单数据，不直接计算复杂权限 | 菜单过滤应放在 `AuthController/AuthState` 或上层服务 |

因此，当前项目适合采用“后端权限真相源 + 前端消费权限快照”的方案。

## 3. 总体方案

推荐方案是标准单租户 `RBAC`：

```text
User
  -> UserRole
  -> Role
  -> RolePermission
  -> Permission
```

权限按用途分类：

| 类型 | 示例编码 | 用途 | 强制边界 |
|---|---|---|---|
| `MENU` | `menu:dashboard` | 控制菜单是否展示 | 前端体验层，后端菜单接口负责过滤 |
| `ACTION` | `system:user:create` | 控制按钮、表单操作、批量动作 | 前端显隐 + 后端业务接口校验 |
| `API` | `api:users:create` | 控制具体后端接口访问 | 后端 `Guard` 强校验 |

核心原则：

- 菜单权限解决“看不看得到入口”。
- 操作权限解决“页面里能不能点某个动作”。
- API 权限解决“请求到后端是否允许执行”。
- 同一个业务动作应同时具备稳定的 `ACTION` 权限和 `API` 权限。
- 前端权限不足时隐藏或禁用入口，后端权限不足时返回 `FORBIDDEN`。

## 4. 角色模型

首版建议内置以下角色：

| 角色编码 | 中文名称 | 定位 |
|---|---|---|
| `SUPER_ADMIN` | 超级管理员 | 拥有全部系统能力，只保留给系统维护者 |
| `ADMIN` | 管理员 | 管理业务配置、普通用户和主要数据 |
| `OPERATOR` | 操作员 | 处理日常业务数据 |
| `VIEWER` | 只读用户 | 只允许查看授权范围内的数据 |

角色设计要求：

- 角色编码使用英文大写下划线。
- 角色名称面向用户显示，使用中文。
- 角色是否启用由 `isActive` 控制，禁用角色后用户不能通过该角色获得权限。
- `SUPER_ADMIN` 是系统保底角色，但实现时仍建议通过权限集合计算，而不是到处写死分支。
- 访问控制不硬编码 `SUPER_ADMIN` 放行；seed 应给 `SUPER_ADMIN` 绑定全部 `isActive = true` 的权限。
- 新增系统权限后，seed 必须自动补齐给 `SUPER_ADMIN`。
- “至少保留一个有效超级管理员”属于安全保护规则，可以单独识别 `SUPER_ADMIN`。
- 账号为 `ACTIVE` 但没有任何启用角色时，首版应拒绝登录并返回 `NO_ACTIVE_ROLE`。

## 5. 权限编码规范

权限编码必须稳定、可读、可检索，不使用中文和展示文案作为权限主键。

### 5.1 菜单权限

格式：

```text
menu:<menuKey>
```

建议首批菜单权限：

| 菜单 | 路由 | 权限编码 |
|---|---|---|
| 工作台 | `/` | `menu:dashboard` |
| 基础数据 | `/base-data` | `menu:base-data` |
| 系统设置 | `/settings` | `menu:settings` |

首版推荐把用户管理、角色管理、权限管理放在 `系统设置` 页面内部，用 Tabs 展示，不作为侧边栏二级菜单。这样首版只需要 `/settings` 一个路由和 `menu:settings` 一个菜单权限。

Tabs 的可见性不使用 `menu:*` 权限，而使用对应的操作权限：

| Tab | 所属页面 | 可见性权限 |
|---|---|---|
| 用户管理 | `/settings` | `system:user:list` |
| 角色管理 | `/settings` | `system:role:list` |
| 权限管理 | `/settings` | `system:permission:list` |

只有当后续明确需要独立深链页面或侧边栏二级菜单时，才考虑为这些管理功能新增独立子路由和对应的菜单权限。

### 5.2 操作权限

格式：

```text
<domain>:<resource>:<action>
```

示例：

| 场景 | 权限编码 |
|---|---|
| 查看用户列表 | `system:user:list` |
| 创建用户 | `system:user:create` |
| 编辑用户 | `system:user:update` |
| 禁用用户 | `system:user:disable` |
| 重置密码 | `system:user:reset-password` |
| 分配角色 | `system:user:assign-roles` |
| 查看角色 | `system:role:list` |
| 编辑角色权限 | `system:role:assign-permissions` |
| 查看权限 | `system:permission:list` |

### 5.3 API 权限

格式：

```text
api:<resource>:<action>
```

示例：

| 接口 | 权限编码 |
|---|---|
| `GET /api/users` | `api:users:list` |
| `POST /api/users` | `api:users:create` |
| `GET /api/users/:id` | `api:users:detail` |
| `PATCH /api/users/:id` | `api:users:update` |
| `PATCH /api/users/:id/status` | `api:users:update-status` |
| `POST /api/users/:id/reset-password` | `api:users:reset-password` |
| `PUT /api/users/:id/roles` | `api:users:assign-roles` |
| `GET /api/roles` | `api:roles:list` |
| `PUT /api/roles/:id/permissions` | `api:roles:assign-permissions` |

`ACTION` 与 `API` 权限需要在 seed 中维护映射，避免前端按钮权限和后端接口权限脱节。

| 业务动作 | ACTION 权限 | API 权限 |
|---|---|---|
| 创建用户 | `system:user:create` | `api:users:create` |
| 编辑用户 | `system:user:update` | `api:users:update` |
| 禁用用户 | `system:user:disable` | `api:users:update-status` |
| 重置密码 | `system:user:reset-password` | `api:users:reset-password` |
| 分配角色 | `system:user:assign-roles` | `api:users:assign-roles` |
| 编辑角色权限 | `system:role:assign-permissions` | `api:roles:assign-permissions` |

## 6. 数据库设计建议

后台权限体系首版采用混合主键策略：`users.id` 使用 `UUID`，`roles.id` 和 `permissions.id` 使用 `Int` 自增。原因是用户 ID 会进入 Token、审计日志、前端状态和部分 API 路径，使用 `UUID` 可以降低枚举风险；角色和权限属于系统配置数据，数量小、内部管理为主，使用 `Int` 更简单、更轻量。

主键约束：

- `users.id`：使用 `String @id @default(uuid()) @db.Uuid`。
- `roles.id`：使用 `Int @id @default(autoincrement())`。
- `permissions.id`：使用 `Int @id @default(autoincrement())`。
- 所有引用 `users.id` 的外键字段使用 `String @db.Uuid`。
- 所有引用 `roles.id`、`permissions.id` 的外键字段使用 `Int`。
- 业务单据编号、订单号、供应商编号等仍应使用独立业务编号字段，不直接暴露数据库主键作为业务编号。

Prisma 命名约束：

- Prisma Model 使用 `PascalCase`，例如 `User`、`UserProfile`、`RolePermission`。
- 数据库表名使用 `snake_case` 复数形式，例如 `users`、`user_profiles`、`role_permissions`。
- 字段在 TypeScript/Prisma 中使用 `camelCase`，数据库列名使用 `snake_case`。
- 通过 `@map` 和 `@@map` 维护映射，避免 Prisma Client 调用变得别扭。

示例：

```prisma
model User {
  id           String   @id @default(uuid()) @db.Uuid
  username     String   @unique
  passwordHash String   @map("password_hash")
  tokenVersion Int      @default(0) @map("token_version")
  createdAt    DateTime @default(now()) @map("created_at")
  updatedAt    DateTime @updatedAt @map("updated_at")

  @@map("users")
}
```

首版删除策略：

- 用户不提供物理删除，只允许 `DISABLED` 或 `LOCKED`。
- 角色不提供物理删除，只允许 `isActive = false`；系统内置角色禁止禁用。
- 权限不提供后台删除；系统权限由 seed 管理，变更需通过迁移或受控脚本。
- 用户角色关系和角色权限关系允许通过授权操作变更，但必须写入审计日志。
- 授权操作人字段引用 `users.id`，如果未来确实发生用户物理删除，应使用 `ON DELETE SET NULL` 保留历史记录。

### 6.1 users

账号主表，只保存登录和账号状态相关字段。

| 字段 | 类型建议 | 说明 |
|---|---|---|
| `id` | `String @db.Uuid` | 主键，默认 `uuid()` |
| `username` | `String` | 登录账号，唯一 |
| `passwordHash` | `String` | 密码哈希，不返回前端 |
| `status` | `UserStatus` | `ACTIVE`、`DISABLED`、`LOCKED` |
| `tokenVersion` | `Int` | 用于强制旧 Token 失效 |
| `lastLoginAt` | `DateTime?` | 最近登录时间 |
| `createdAt` | `DateTime` | 创建时间 |
| `updatedAt` | `DateTime` | 更新时间 |

索引：

- `users.username` 唯一。
- `users.status` 普通索引，便于后台筛选。

### 6.2 user_profiles

用户资料表，与账号一对一。

| 字段 | 类型建议 | 说明 |
|---|---|---|
| `id` | `Int` | 主键 |
| `userId` | `String @db.Uuid` | 关联 `users.id`，唯一 |
| `displayName` | `String` | 用户显示名 |
| `phone` | `String?` | 手机号，可唯一 |
| `email` | `String?` | 邮箱，可唯一 |
| `avatarUrl` | `String?` | 头像地址 |
| `departmentName` | `String?` | 首版可先用文本，后续组织架构再独立建表 |
| `postName` | `String?` | 岗位名称 |
| `createdAt` | `DateTime` | 创建时间 |
| `updatedAt` | `DateTime` | 更新时间 |

首版 `username` 必须唯一。`phone`、`email` 如果不作为登录凭证，先不强制唯一；如后续需要唯一，优先使用 PostgreSQL 部分唯一索引，仅约束非空值。

### 6.3 roles

角色表。

| 字段 | 类型建议 | 说明 |
|---|---|---|
| `id` | `Int` | 主键 |
| `code` | `String` | 角色编码，唯一 |
| `name` | `String` | 中文名称 |
| `description` | `String?` | 说明 |
| `isSystem` | `Boolean` | 系统内置角色，避免被误删 |
| `isActive` | `Boolean` | 是否启用 |
| `createdAt` | `DateTime` | 创建时间 |
| `updatedAt` | `DateTime` | 更新时间 |

### 6.4 user_roles

用户角色关系表，一个用户可以绑定多个角色。

| 字段 | 类型建议 | 说明 |
|---|---|---|
| `userId` | `String @db.Uuid` | 关联用户 |
| `roleId` | `Int` | 关联角色 |
| `assignedAt` | `DateTime` | 授权时间 |
| `assignedBy` | `String @db.Uuid?` | 授权操作人 |

约束：

- 组合主键：`[userId, roleId]`。
- 首版不做用户物理删除，禁用用户不删除关系。
- 删除系统内置角色应被业务层禁止。

### 6.5 permissions

权限定义表。

| 字段 | 类型建议 | 说明 |
|---|---|---|
| `id` | `Int` | 主键 |
| `code` | `String` | 权限编码，唯一 |
| `name` | `String` | 中文名称 |
| `type` | `PermissionType` | `MENU`、`ACTION`、`API` |
| `module` | `String` | 模块，例如 `system`、`base-data` |
| `description` | `String?` | 说明 |
| `sortOrder` | `Int` | 展示排序 |
| `isSystem` | `Boolean` | 系统内置权限 |
| `isActive` | `Boolean` | 是否启用 |
| `createdAt` | `DateTime` | 创建时间 |
| `updatedAt` | `DateTime` | 更新时间 |

`module` 仅用于权限树分组、查询过滤和后台展示，不参与权限判定。命名使用英文小写短横线或小写英文单词，不使用中文。

### 6.6 role_permissions

角色权限关系表。

| 字段 | 类型建议 | 说明 |
|---|---|---|
| `roleId` | `Int` | 关联角色 |
| `permissionId` | `Int` | 关联权限 |
| `assignedAt` | `DateTime` | 授权时间 |
| `assignedBy` | `String @db.Uuid?` | 授权操作人 |

约束：

- 组合主键：`[roleId, permissionId]`。
- 只允许绑定 `isActive = true` 的权限。

### 6.7 refresh_sessions

用于 refresh token 管理和安全审计。

| 字段 | 类型建议 | 说明 |
|---|---|---|
| `id` | `String` | 会话主键，建议 `cuid()` |
| `userId` | `String @db.Uuid` | 关联用户 |
| `refreshTokenHash` | `String` | refresh token 哈希 |
| `userAgent` | `String?` | 客户端信息 |
| `ipAddress` | `String?` | 登录 IP |
| `expiresAt` | `DateTime` | 过期时间 |
| `revokedAt` | `DateTime?` | 撤销时间 |
| `createdAt` | `DateTime` | 创建时间 |

### 6.8 audit_logs

后台安全相关操作建议记录审计日志。

| 字段 | 类型建议 | 说明 |
|---|---|---|
| `id` | `Int` | 主键 |
| `actorUserId` | `String @db.Uuid?` | 操作人 |
| `action` | `String` | 操作类型 |
| `resourceType` | `String` | 资源类型 |
| `resourceId` | `String?` | 资源 ID |
| `metadata` | `Json?` | 脱敏后的附加信息 |
| `ipAddress` | `String?` | IP |
| `userAgent` | `String?` | 客户端信息 |
| `createdAt` | `DateTime` | 创建时间 |

审计日志禁止保存明文密码、Token、数据库连接串和其他敏感配置。

`metadata` 必须按白名单写入，禁止保存明文密码、密码哈希、access token、refresh token、refresh token hash、数据库连接串、完整请求头、完整请求体以及强隐私数据。允许保存脱敏后的目标账号、变更字段、旧状态和新状态。

## 8. 认证与授权流程

### 8.1 登录流程

```text
POST /api/auth/login
  -> 校验 username/password
  -> 校验用户 status
  -> 校验用户至少有一个启用角色
  -> 查询用户角色和权限
  -> 写入 refresh_sessions
  -> 返回 accessToken、refreshToken、user、roles、permissions、menus
```

登录响应建议：

```ts
type LoginResponse = {
  accessToken: string;
  refreshToken: string;
  user: {
    id: string;
    username: string;
    displayName: string;
    avatarUrl?: string;
  };
  roles: Array<{
    code: string;
    name: string;
  }>;
  permissions: string[];
  menus: MenuNode[];
};
```

菜单节点结构：

```ts
type MenuNode = {
  key: string;
  title: string;
  route?: string;
  icon?: string;
  permissionCode: string;
  sortOrder: number;
  children?: MenuNode[];
};
```

注意：

- `passwordHash`、`refreshTokenHash`、`tokenVersion` 不返回前端。
- `permissions` 返回权限编码列表，前端用 `Set<String>` 缓存。
- `menus` 由后端按权限过滤后返回，前端不自己拼完整菜单树。
- `menus` 首版最多返回两级节点。
- 登录接口直接返回 `permissions` 和 `menus`，用于桌面端登录后立即进入 Shell。
- 另提供 `GET /api/auth/session`，用于应用启动恢复登录态时重新获取用户、角色、权限和菜单快照。

### 8.2 Access Token

`accessToken` 建议包含最小必要信息：

```ts
type JwtPayload = {
  sub: string;
  username: string;
  tokenVersion: number;
};
```

不建议把全部权限编码直接塞进 `accessToken`，原因是：

- 权限列表可能增长。
- 角色权限变更后 Token 内权限会过期。
- 后端接口校验需要以当前数据库权限为准，至少应具备可刷新机制。

首版可以在登录时返回权限快照给前端用于 UI，后端 `Guard` 查询缓存或数据库完成强校验。

`JwtAuthGuard` 每次必须校验当前用户状态和 `tokenVersion`。当 `payload.tokenVersion !== user.tokenVersion` 时，返回 `401 TOKEN_REVOKED` 或 `401 TOKEN_EXPIRED`。如果 Guard 只校验 JWT 签名而不查用户状态和版本，`tokenVersion` 不具备实际失效能力。

### 8.3 接口授权流程

接口使用装饰器声明所需权限：

```ts
@ApiPermission('api:users:create')
@Post()
createUser() {}
```

请求进入后：

```text
JwtAuthGuard
  -> 解析 accessToken
  -> 校验用户存在、状态正常、tokenVersion 有效
  -> ApiPermissionGuard
  -> 读取接口元数据中的权限编码
  -> 校验当前用户角色是否拥有该 API 权限
  -> 放行或返回 FORBIDDEN
```

建议错误码：

| 场景 | HTTP 状态 | `code` |
|---|---|---|
| 未登录 | `401` | `UNAUTHORIZED` |
| Token 过期 | `401` | `TOKEN_EXPIRED` |
| Token 已撤销 | `401` | `TOKEN_REVOKED` |
| 用户禁用 | `403` | `USER_DISABLED` |
| 无启用角色 | `403` | `NO_ACTIVE_ROLE` |
| 权限不足 | `403` | `FORBIDDEN` |
| 角色禁用 | `403` | `ROLE_DISABLED` |

### 8.4 Refresh Token 轮换

refresh token 只保存哈希。首版采用“原记录更新 hash”的轮换方式：

```text
POST /api/auth/refresh
  -> 校验 refresh token 格式
  -> 查询 refresh_sessions
  -> 比对 refreshTokenHash
  -> 校验未过期、未撤销
  -> 生成新的 accessToken 和 refreshToken
  -> 更新 refresh_sessions.refreshTokenHash
  -> 返回新 token
```

每次刷新后旧 refresh token 立即失效。退出登录时设置 `revokedAt`。如果检测到已撤销或已轮换的 refresh token 被再次使用，应拒绝刷新并记录安全审计日志。

### 8.5 权限读取策略

首版后端不启用权限缓存，`ApiPermissionGuard` 每次从数据库读取当前用户、账号状态、`tokenVersion`、启用角色和启用权限后完成接口权限校验。

这样首版可以先避免缓存失效复杂度：

- 用户角色变更后，事务提交后的下一次接口鉴权立即使用最新角色。
- 角色权限变更后，事务提交后的下一次接口鉴权立即使用最新权限。
- 角色禁用、权限禁用、用户禁用或锁定后，下一次接口鉴权立即生效。
- `/api/auth/session`、`/api/auth/permissions`、`/api/menus/me` 均从数据库读取最新权限快照。

如果后续出现明确性能瓶颈，再单独设计权限缓存。届时缓存 key 可使用 `authz:<userId>:<tokenVersion>`，并必须补充用户角色变更、角色权限变更、角色禁用、权限禁用和 `tokenVersion` 变化后的失效策略。

### 8.6 账号锁定策略

`LOCKED` 表示账号处于安全锁定状态，不是物理删除，也不是角色被禁用。账号被锁定后：

- 不允许登录。
- 已有 access token 应通过递增 `tokenVersion` 失效。
- 用户资料、角色关系和审计记录保留。
- 管理员可以根据业务流程将账号恢复为 `ACTIVE`。

`LOCKED` 与 `DISABLED` 的区别：

| 状态 | 含义 | 典型场景 | 是否允许登录 |
|---|---|---|---|
| `ACTIVE` | 正常启用 | 正常用户 | 是 |
| `DISABLED` | 业务禁用 | 离职、账号停用、长期关闭 | 否 |
| `LOCKED` | 安全锁定 | 账号异常、临时冻结、后续登录失败次数过多自动锁定 | 否 |

首版仅支持管理员手动设置 `LOCKED`，不实现登录失败次数自动锁定。后续如果要做自动锁定，再新增 `failedLoginAttempts`、`lockedUntil`、`lastFailedLoginAt` 等字段。

## 9. 菜单与前端权限接入

前端已有 `SidebarMenuItemConfig.permissionCode`，后续应保持该边界：

- `Sidebar` 只渲染传入的菜单数据。
- `Sidebar` 不直接请求权限接口。
- `Sidebar` 不直接计算复杂权限。
- 登录态、权限集合和菜单数据由上层 `AuthController` 或 `ShellController` 管理。

推荐前端目录：

```text
frontend/lib/
  modules/
    auth/
      auth_controller.dart
      auth_repository.dart
      login_page.dart
      models/
    system/
      users/
      roles/
      permissions/
  shared/
    services/
      api_client.dart
      token_storage.dart
```

推荐前端权限判断：

```dart
class AuthState {
  AuthState({required this.permissions});

  final Set<String> permissions;

  bool can(String code) => permissions.contains(code);

  bool canAny(List<String> codes) {
    return codes.any((code) => permissions.contains(code));
  }

  bool canAll(List<String> codes) {
    return codes.every((code) => permissions.contains(code));
  }
}
```

菜单处理：

```text
后端返回 menus
  -> AuthController 保存菜单树
  -> AppShell 将菜单树转换为 SidebarMenuItemConfig
  -> Sidebar 渲染
```

系统设置页面内部 Tabs 处理：

```text
进入 /settings
  -> SettingsController 读取 AuthState
  -> can('system:user:list') 决定是否展示用户管理 Tab
  -> can('system:role:list') 决定是否展示角色管理 Tab
  -> can('system:permission:list') 决定是否展示权限管理 Tab
```

如果当前用户拥有 `menu:settings`，但没有任何系统设置 Tab 的 `list` 权限，前端应显示空状态或无权限提示，不应展示空白页面。

按钮处理：

```text
页面 Controller 持有 AuthState
  -> can('system:user:create')
  -> 控制按钮显示或禁用
```

API 请求处理：

- `ApiClient` 统一添加 `Authorization: Bearer <accessToken>`。
- 收到 `401 TOKEN_EXPIRED` 时尝试刷新 Token。
- 收到 `401 TOKEN_REVOKED` 时清理登录态并回到登录页。
- 收到 `403 USER_DISABLED`、`403 ROLE_DISABLED`、`403 NO_ACTIVE_ROLE` 时清理权限快照，并提示账号或角色不可用。
- 收到 `403 FORBIDDEN` 时提示“无权限执行此操作”，不要重复重试。
- refresh token 轮换后，`ApiClient` 必须实现单飞刷新机制，多个请求同时收到 `401` 时只触发一次 refresh，其余请求等待同一个 refresh Future。
- refresh 失败后统一退出登录，避免多个请求分别改写登录态。

Token 存储：

| 模式 | access token | refresh token |
|---|---|---|
| 首版记住登录 | 内存保存；应用启动时使用 refresh token 换取新的 access token | 使用系统安全存储持久化 |
| 当前会话 | 内存 | 内存或系统安全存储中的当前 refresh token |
| 退出登录 | 清理内存 access token | 删除系统安全存储中的 refresh token，并调用退出接口撤销 refresh session |

首版需要支持“记住登录”，因此必须接入 Flutter Desktop 系统安全存储方案。`refreshToken` 不允许保存到普通文件、日志、SharedPreferences 或明文配置中。系统安全存储至少需要覆盖目标桌面平台，例如 macOS Keychain 和 Windows Credential Manager。

## 12. 安全规则

必须遵守：

- 密码使用 `argon2` 或 `bcrypt` 哈希，不保存明文密码。
- 密码哈希优先使用 `argon2id`；如果部署环境对原生依赖不友好，再使用 `bcrypt`。
- refresh token 只保存哈希。
- 日志中不输出密码、Token、数据库连接串、用户隐私数据。
- 禁用用户后应立即使旧 Token 失效。
- 重置密码应递增 `tokenVersion`，让旧 Token 失效。
- 首版权限校验每次查库，不启用权限缓存；编辑角色权限后，后续接口校验必须使用数据库中的最新权限。
- 删除系统内置角色和系统内置权限应被禁止。
- `SUPER_ADMIN` 至少保留一个有效用户，避免系统被锁死。
- 用户状态变更、用户角色分配、角色权限分配、重置密码必须使用数据库事务，并在同一事务或可靠后置流程中写入审计日志。
- `PATCH /api/users/:id/status` 只能接收 `ACTIVE`、`DISABLED`、`LOCKED`，不能隐式接受任意字符串。
- 禁用、锁定用户时必须递增 `tokenVersion`；启用用户不降低 `tokenVersion`。
- 首版至少保护以下场景：禁止禁用最后一个有效 `SUPER_ADMIN` 用户，禁止移除最后一个有效 `SUPER_ADMIN` 角色绑定，禁止禁用 `SUPER_ADMIN` 角色，禁止删除系统内置角色。

## 13. 验收标准

### 13.1 需求验收

使用 EARS 描述核心验收条件：

1. 当用户提交正确账号密码且账号状态为 `ACTIVE` 时，系统应返回登录 Token、用户资料、角色列表、权限编码集合和可见菜单。
2. 当用户账号状态为 `DISABLED` 或 `LOCKED` 时，系统应拒绝登录并返回稳定错误码。
3. 当用户访问未授权 API 时，系统应返回 `403 FORBIDDEN`，并且不得执行目标业务动作。
4. 当角色权限被调整后，系统应确保事务提交后的后续接口校验使用最新权限。
5. 当用户没有某个菜单权限时，系统应不向前端返回该菜单节点。
6. 当用户进入 `/settings` 且缺少某个管理 Tab 的 `list` 权限时，系统应隐藏对应 Tab。
7. 当前端收到 `FORBIDDEN` 时，系统应显示无权限提示，不应继续重试同一请求。
8. 当管理员重置用户密码或禁用用户时，系统应使该用户旧 Token 失效。
9. 当系统内置角色或权限被删除时，系统应拒绝该操作。
10. 当用户没有任何启用角色时，系统应拒绝登录并返回 `NO_ACTIVE_ROLE`。
11. 当 refresh token 被使用刷新后，旧 refresh token 应立即失效。
12. 当角色权限变更后，相关用户后续接口校验应从数据库读取并使用最新权限。
13. 当操作会导致系统不存在有效 `SUPER_ADMIN` 时，系统应拒绝该操作。

### 13.2 技术验收

- `backend/prisma/schema.prisma` 中存在用户、角色、权限和关系表。
- 所有新增 DTO 使用 `class-validator` 校验。
- 所有受保护接口通过 `JwtAuthGuard`。
- 所有需要权限的接口声明 `@ApiPermission(...)`。
- 前端 `ApiClient` 统一注入 Token。
- 前端菜单来自后端当前用户菜单接口，不再使用静态默认值作为真实菜单。
- 系统设置页面内部用户、角色、权限 Tabs 使用 `system:*:list` 操作权限控制，不依赖额外子菜单权限。
- OpenAPI 能导出登录、会话、用户、角色、权限接口。
- `JwtAuthGuard` 校验用户状态和 `tokenVersion`。
- 首版权限校验每次查库；角色权限变更后旧权限不能继续生效。

| refresh token | 每次刷新都轮换 |
| 权限读取 | 首版每次查库，不启用权限缓存 |
| 角色权限变更 | 因首版不启用缓存，事务提交后的下一次鉴权从数据库读取最新权限；后续如引入缓存，再单独设计失效策略 |
| Flutter Desktop token | 首版需要支持“记住登录”，必须使用系统安全存储持久化 `refreshToken` |
| 开发管理员 | 本地开发管理员账号和密码均为 `admin`，仅开发环境允许 |
| 生产管理员 | 生产环境禁止固定默认管理员密码，生产管理员密码必须在部署或首次初始化时重新设置 |
| `LOCKED` 状态 | 首版作为管理员手动设置的安全锁定状态，不实现登录失败次数自动锁定 |

后续如果要新增权限缓存、登录失败自动锁定、用户物理删除或多租户能力，必须重新进入专项设计确认。
