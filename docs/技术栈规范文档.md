# QSDMS_DESKTOP 项目技术栈文档

> 适用项目：QSDMS-企业数据管理系统（桌面端）  
> 目标平台：macOS、Windows  
> 技术方向：Flutter Desktop + NestJS + PostgreSQL

## 1. 项目定位

`QSDMS_DESKTOP` 定位为桌面优先的企业数据管理系统，用于内部运营人员处理订单、供应商、产品、仓储、发运、财务等高频业务流程。

项目不是移动端 App 的桌面放大版，而是面向鼠标、键盘、大屏、多窗口工作习惯设计的企业工作台。核心体验应围绕：

- 高密度数据表格
- 快速筛选与批量操作
- 订单详情联动
- 异常优先处理
- 表单录入与校验
- 操作反馈与审计追踪
- macOS / Windows 一致分发

## 2. 总体架构

推荐采用前后端分离架构：

```text
Flutter Desktop Client
  -> REST API / OpenAPI Contract
  -> NestJS Backend
  -> PostgreSQL
```

前端负责桌面端交互、页面状态、表格操作、表单校验、接口调用和本地体验优化。后端负责业务规则、权限控制、数据持久化、审计日志、任务调度和对外接口契约。

项目第一阶段应优先采用模块化单体架构，不提前拆分微服务。只有当业务边界、部署边界、团队边界都足够清晰后，才考虑拆分独立服务。

## 3. 前端技术栈

### 3.1 核心框架

前端使用 `Flutter Desktop + GetX`。

- `Flutter`：作为跨平台桌面 UI 框架，目标平台为 `macOS` 和 `Windows`。
- `Dart`：作为前端主要开发语言。
- `GetX`：作为状态管理、路由管理和依赖注入的统一方案。

前端代码应围绕桌面端工作台体验组织，不按移动端页面思维简单放大。页面设计优先支持大屏、高密度表格、键盘操作、鼠标悬停、快速筛选、批量操作和详情联动。

### 3.2 GetX 使用边界

`GetX` 在本项目中承担三类职责：

- 状态管理：使用 `GetxController` 管理页面状态、查询条件、表单状态和接口加载状态。
- 路由管理：使用 `GetPage` 维护页面路由、参数传递和中间件。
- 依赖注入：使用 `Bindings` 注入 `Controller`、`Service` 和 `Repository`。

约束：

- `Widget` 中不直接写复杂业务逻辑。
- `Controller` 负责页面编排，不直接拼接 SQL 或承载后端业务规则。
- API 调用统一下沉到 `Service` 或 `Repository`，避免分散在页面组件中。
- 路由名称、API 路径、DTO 字段名等技术资产使用英文。
- 用户可见文案默认使用中文。

### 3.3 推荐前端生态

- HTTP 客户端：`dio`
- 数据模型：`json_serializable` / `freezed`
- 本地缓存：按需使用轻量本地存储；不得把服务端权威业务数据长期只保存在本地
- 日志：封装统一前端日志入口，禁止输出敏感信息
- 环境配置：按 `dev`、`test`、`prod` 区分 API 地址和运行参数

### 3.4 前端目录建议

```text
frontend/
  lib/
    app/
      routes/
      bindings/
      theme/
    modules/
    shared/
      widgets/
      services/
      repositories/
      models/
      utils/
```

目录按业务模块组织。公共组件、公共请求封装、公共模型和工具函数放入 `shared/`，避免业务模块之间互相穿透引用。

## 4. 后端技术栈

### 4.1 核心框架

后端使用 `NestJS + PostgreSQL`。

- `NestJS`：作为后端应用框架。
- `TypeScript`：作为后端主要开发语言。
- `PostgreSQL`：作为主业务数据库。
- `Node.js`：使用项目锁定版本，不在代码中依赖本机隐式版本。

后端优先采用模块化单体结构。按业务域拆分 `Module`

### 4.2 推荐后端生态

- ORM / 数据访问：`Prisma`
- 数据库迁移：`Prisma Migrate`
- API 文档：`@nestjs/swagger`
- 参数校验：`class-validator`、`class-transformer`
- 配置管理：`@nestjs/config`
- 鉴权认证：`Passport`、`JWT`、`Refresh Token`
- 密码处理：成熟哈希库，例如 `argon2` 或 `bcrypt`
- 日志：`pino` 或 `winston`
- 定时任务：`@nestjs/schedule`
- 队列任务：按需引入 `Redis + BullMQ`
- 测试：`Jest`、`Supertest`

`Redis`、队列、对象存储、搜索引擎等能力按业务压力逐步引入，不作为第一阶段默认复杂化前提。

### 4.3 PostgreSQL 使用规范

`PostgreSQL` 是服务端业务数据真相源。

- 所有核心业务数据以数据库为准，前端缓存只作为体验优化。
- 数据库 schema 变更必须通过迁移管理，不手工改生产库结构。
- 表名、字段名、索引名使用英文，语义清晰。
- 业务编号不直接等同数据库自增主键，应通过单独规则生成。
- 常用查询条件必须评估索引。
- 涉及金额、库存、状态流转、审计日志的更新必须在后端保证一致性。

## 5. 接口与数据契约

前后端通过 `REST API` 通信，接口契约以 `OpenAPI` 文档为准。

- 后端 DTO 是接口字段的权威定义。
- 前端模型应从接口契约派生或与接口契约保持一致。
- API 路径、请求字段、响应字段、错误码使用英文。
- 用户提示、表单校验消息、业务说明使用中文。
- 分页、排序、筛选、批量操作应形成统一协议。

建议统一响应结构：

```ts
type ApiResponse<T> = {
  code: string;
  message: string;
  data: T;
  requestId: string;
};
```

错误码应稳定、可检索、可用于前端分支处理。不得把数据库错误、堆栈信息或敏感配置直接返回给前端。

## 6. 测试与质量要求

项目采用 TDD 开发模式。修复 Bug 时应优先补充失败测试，再实现修复。

前端验证重点：

- `Controller` 状态变化
- 表单校验
- API 异常处理
- 路由跳转
- 关键业务页面的桌面端布局

后端验证重点：

- `Service` 业务规则
- `Controller` 请求校验
- 权限与鉴权逻辑
- 数据库迁移
- 关键 API 的集成测试

每次声称完成前必须运行与改动范围匹配的验证命令。命令失败或测试失败时，必须明确报告失败结果，不得掩盖。

## 7. 安全与配置规范

- 不硬编码密钥。
- 不提交 `.env`。
- 不在日志中输出密码、Token、数据库连接串、用户隐私数据。
- 所有环境变量通过配置模块读取。
- 生产环境必须启用明确的 CORS、限流、鉴权和日志脱敏策略。
- 修改公共 API、数据结构、数据库 schema 或删除文件前，必须先获得确认。

## 8. Git 与工程协作

- Commit Message 使用英文类型加中文描述，例如 `docs: 补充前后端技术栈规范`。
- 小步提交，每次功能改动后提交对应范围。
- 不把格式化、无关重构和业务变更混在同一次提交里。
- 修改技术栈、公共契约、数据库 schema、权限模型时，应同步更新项目文档。

## 9. 当前技术栈结论

本项目技术栈结论如下：

```text
Frontend: Flutter Desktop + Dart + GetX + dio
Backend: NestJS + TypeScript + PostgreSQL + Prisma
Contract: REST API + OpenAPI
Testing: Flutter test + Jest + Supertest
Architecture: Modular Monolith
```

除非后续有明确业务约束变化，本项目不再反复评估基础框架选型，后续工作应围绕以上技术栈推进实现。
